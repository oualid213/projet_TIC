Chapitre 5: Système de Gestion de Version

cette section a été faite par abdelhamid toubache

Ce chapitre constitue une introduction à Git et GitHub, les deux systèmes de gestion de version les plus populaires parmi les développeurs. Voici un résumé des points clés abordés dans chaque section :

  1. Introduction :

    Ce chapitre explore l'importance de Git et GitHub dans le développement de logiciels et définit leur utilité. Git, un outil de gestion de versions créé par Linus Torvalds en 2005, permet de conserver un historique détaillé des modifications apportées à un projet. En utilisant des logiciels de gestion de versions, les équipes peuvent collaborer de manière plus efficace et améliorer leur productivité.

  2. L'utilité concrète d'un système de gestion de version :

    Dans cette section, nous explorons l'importance d'un système de gestion de versions en utilisant un site web comme exemple. Lorsque plusieurs développeurs travaillent sur un projet commun, il devient crucial de coordonner leurs modifications, d'éviter les conflits et de garantir que chaque développeur dispose de la version la plus récente du projet. Un système de gestion de versions offre une solution efficace pour gérer ces aspects de manière optimale.

  3. Les deux modèles des logiciels de gestion de version :

    Dans cette section, nous abordons les deux modèles de gestion de versions les plus répandus : le modèle centralisé et le modèle décentralisé. Dans le modèle centralisé, le code source est stocké sur un serveur central, tandis que dans le modèle décentralisé, chaque utilisateur possède une copie complète du code source sur sa propre machine. Git adopte le modèle décentralisé, ce qui confère une plus grande flexibilité et une meilleure sécurité.

  4. Qu'est-ce que GitHub ?

    Git est un logiciel de gestion de version extrêmement puissant, tandis que GitHub, quant à lui, est un service en ligne renommé qui offre la possibilité d'héberger des dépôts Git. En réalité, GitHub se distingue en tant que premier et plus vaste hébergeur de dépôts Git à l'échelle mondiale. Les développeurs, quant à eux, peuvent aisément partager leur code sur cette plateforme et collaborer avec d'autres en proposant et en ajoutant des fonctionnalités supplémentaires. Grâce à cette combinaison de Git et de GitHub, les équipes de développement ont l'avantage de bénéficier d'un environnement de travail plus efficace et productif pour leurs projets en commun.

  5. L'utilisation de Git :

    Dans cette section, diverses interfaces pour utiliser Git sont présentées, avec un accent particulier sur l'utilisation de la ligne de commande. L'utilisation de la ligne de commande permet d'accéder à toutes les commandes Git disponibles et assure une expérience cohérente pour tous les utilisateurs, quel que soit leur système d'exploitation ou leur environnement de développement.

  6. Installation de Git:

    Les étapes d'installation de Git sont expliquées, recommandant de télécharger la dernière version à partir du site officiel et de suivre les instructions d'installation par défaut.

  7. Paramétrage de Git:

    Après l'installation, il est nécessaire de configurer Git en fournissant un nom d'utilisateur et une adresse e-mail. Cela peut être fait en utilisant la commande "git config" avec l'option "--global" pour appliquer les informations de configuration à tous les projets Git.

  8. Démarrer un dépôt Git:

    Un dépôt Git est la copie et l'importation de l'ensemble des fichiers d'un projet dans Git. Il existe deux façons de créer un dépôt Git : importer un répertoire existant ou cloner un dépôt déjà existant. Les étapes pour démarrer un dépôt Git sont expliquées.

  9. La gestion des informations selon Git:

    Git gère les informations en prenant des instantanés du contenu de l'espace de travail à chaque validation.

  10. Les états des fichiers:

    Les états des fichiers dans Git sont gérés pour suivre les modifications sur les fichiers d'un projet. Un fichier peut être soit "sous suivi de version" (tracked), ce qui signifie qu'il fait partie du dernier instantané enregistré par Git, soit "non suivi" (untracked), ce qui indique qu'il ne fait pas partie du dernier instantané et n'a pas été indexé.

  11. Les zones du travaille :

    Ces états de fichiers sont associés à trois zones de travail dans Git :

    • Le répertoire de travail correspond à l'extraction des fichiers d'une version spécifique du projet, où l'on peut les utiliser et les modifier.
    • La zone d'index est un fichier situé généralement dans le répertoire Git, qui stocke les informations sur les fichiers qui seront inclus dans le prochain instantané.
    • Le répertoire Git est l'endroit où Git stocke les méta-données et la base de données des objets du projet.

  12. Ajouter ou modifier des fichiers dans un projet et actualiser notre dépôt Git :

    Le processus de travail dans Git consiste à modifier les fichiers dans le répertoire de travail, puis à les indexer en utilisant la commande "git add". Une fois les fichiers indexés, ils peuvent être validés en utilisant la commande "git commit", ce qui les ajoute au répertoire Git.
